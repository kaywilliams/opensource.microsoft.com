Securing the open source supply chain

This is a summary of how we approach securing the open source supply chain at Microsoft.

Microsoft’s software supply chain depends on Open-Source Software (OSS) components at all stages of the developer lifecycle. With Microsoft’s usage of tens of thousands of OSS dependencies in nearly 10 million instances today, it is imperative for us to secure our OSS supply chain as it represents a critical piece of our overall software security posture.

Our approach to securing the OSS supply chain includes both ecosystem and internal initiatives.

Ecosystem

At the ecosystem level, we are working on two efforts: industry standards and open source security.

Industry Standards. Microsoft is active in defining industry standards for supply chain security, working toward an overall system for transmitting software artifacts across end-to-end supply chains, with integrity and policy verification at each step. Our work includes participation in the 3T-SBOM, SPDX, in-toto, TUF, and other security specification communities.

Open Source Security. In August 2020, Microsoft joined with others in the industry to form the Open Source Security Foundation (OpenSSF). The aim of the OpenSSF is to pool resources, including people and direct contributions, to design and deliver ecosystem-level solutions for OSS security. Efforts within the OpenSSF include the following:

  • Identifying Security Threats - security metrics for open-source projects
  • Security Tooling - state of the art, globally accessible security tools
  • Best Practices - awareness and education of security best practices
  • Vulnerability Disclosures – efficient vulnerability reporting and remediation
  • Digital Identity Attestation – ensuring the provenance of open source code
  • Securing Critical Projects – hands-on help for critical open source projects

We're excited to work with the OpenSSF in collaborating to secure the open-source ecosystem and encourage others to get involved.

Internal

Microsoft’s goal to secure internal open-source consumption and to protect Microsoft products and customers against OSS supply chain attacks follows a framework we call the Jobs-to-be-Done (JTBDs).

  1. INGEST IT - ensure we have a saved copy of any external artifact we use for development or in production.

    Threat Scenario: A public package repository becomes compromised, or is unavailable.

    Value Proposition: Teams can still ship any existing assets even if external sources are compromised or unavailable.

  2. REBUILD IT - ensure OSS can be rebuilt from source code.

    Threat Scenario: An attacker gains access to build infrastructure and injects malicious changes that modify the generated binaries.

    Value Proposition: Teams have confidence that the OSS binaries being consumed match the purported source code. This also enables private fix scenarios in the case of zero- day vulnerabilities.

  3. SCAN IT - ensure OSS can be rebuilt from source code.

    Threat Scenario: A team tries to use an OSS package that is malicious (i.e. the eslint scenario) or has vulnerabilities that could lead to a security incident (e.g. apache struts vulnerability in Equifax incident).

    Value Proposition: Teams are alerted if they introduce or are already using vulnerable or malicious OSS.

  4. INVENTORY AND UPDATE IT - ensure we know where external artifacts are deployed in production.

    Threat Scenario: A critical vulnerability is discovered in an OSS package, and our Security Response Center wants to know all the production services using the package to drive response.

    Value Proposition: Teams can prioritize remediation efforts to address the highest risk / “blast radius” issues first.

  5. AUDIT IT - verify provenance of all OSS artifacts and ensure they come through the official supply chain.

    Threat Scenario: A build contains an unknown or insecure OSS dependency (see four risk categories above) or was tampered with between lifecycle phases, either by the original developer or a hostile actor.

    Value Proposition: Teams have confidence that their product only contains trusted dependencies and the code that they have produced.

  6. ENFORCE IT - ensure OSS artifacts are consumed through the official supply chain.

    Threat Scenario: A developer bypasses the official supply chain to consume or update an OSS package with a known vulnerability.

    Value Proposition: Teams and Leadership can rely on secure and trusted OSS consumption within the organization.

  7. PRIVATE FIX IT - ensure all OSS can be privately serviced.

    Threat Scenario: A team with a package dependency is unable to get a critical vulnerability fixed quickly enough by the maintainer.

    Value Proposition: Teams can fix critical vulnerabilities without waiting for public fixes.

Today, we are actively working to implement the complete JTBD framework.
